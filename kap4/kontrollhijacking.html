Kapittel 4 · Kontrollflyt-hijacking

Buffer overflow

Et tilgangskontrollsystem som er korrekt på papir, kan likevel kompromitteres hvis angriperen kan ta over selve programmets kontrollflyt. Buffer overflow er den klassiske angrepsmetoden — fortsatt levende i 2026.

01 · Angriperens mål

Hvorfor kontrollflyt?

All tilgangskontroll bygger på antakelsen om at programmet faktisk gjør det koden sier. Kontrollflyt-hijacking bryter den antakelsen.

En angriper som tar over kontrollflyten i et program, kan utføre hvilken som helst handling programmet selv har rettigheter til. Webserveren leser brukerinput, applikasjonen håndterer en filopplasting, et bibliotek parser en dokumentfil — alle kan være innfallspunkter. Når angriperen får programmet til å hoppe til kode de kontrollerer, er CIA-triaden brutt: konfidensialitet, integritet og tilgjengelighet kompromitteres samtidig. Klassisk buffer overflow er fortsatt blant de vanligste teknikkene, særlig mot C og C++.

  • Morris-ormen (1988): overflow i fingerd på VAX. Spredte seg til ~10 % av internett.
  • CodeRed (2001): overflow i Microsoft IIS. Defacet 359 000 servere på 14 timer.
  • SQL Slammer (2003): 376-byte UDP-pakke mot SQL Server. Doblet seg hvert 8.5 sekund.
  • Heartbleed (2014): read overflow i OpenSSL — lekket private nøkler fra millioner av servere.
  • CVE-2026-20841: Windows RPC Runtime — buffer overflow er fortsatt aktuelt i 2026.

Buffer overflow rammer hovedsakelig C og C++ fordi disse språkene gir programmereren ansvaret for minnehåndtering — ingen runtime sjekker grenser på arrays.

02 · Prosessminne

Linux-prosessens minneoppsett

Stack vokser ned, heap vokser opp — og angriperen jakter på det som ligger i mellom.

En Linux-prosess har minnet delt opp i seksjoner med klare roller. Tekstsegmentet er kompilert kode, skrivebeskyttet. Datasegmentet og BSS holder globale og statiske variabler. Heapen vokser oppover under kjøring (malloc, new). Stacken vokser nedover, og holder funksjonsrammer med argumenter, lokale variabler, rammepeker og returadresse. Det er denne nedoverveksten som gjør stack overflow farlig: en lokal buffer som overflødes, kan overskrive returadressen som ligger over.

03 · Stack-rammer

Hvordan en funksjonskall ser ut i minnet

Hvert funksjonskall pusher en ramme på stacken. Returadressen er det mest interessante feltet for en angriper.

Når func1 kaller func2, allokeres en ny ramme på stacken under func1s. Rammen inneholder argumentene, returadressen (hvor CPU-en hopper når func2 returnerer), forrige rammepeker, eventuelt en exception handler, og lokale variabler. Lokale variabler — inkludert buffere — ligger lavest i rammen. Returadressen ligger høyere oppe. Hvis en lokal buffer overflødes, vokser den oppover gjennom rammen og kan overskrive alt som ligger over.

04 · Stack overflow

Hva skjer ved overflow?

Problemet sitter i C-funksjoner som ikke sjekker lengde — strcpy, strcat, gets.

strcpy(buf, arg2) kopierer fra arg2 til buf til den finner en null-byte. Ingen sjekk på om buf er stor nok. Hvis arg2 er 200 bytes og buf er 128, skrives 72 bytes inn i minnet rett etter bufferen — som inkluderer andre lokale variabler, exception handler og returadresse.

Tre typer skade en angriper kan gjøre:

  1. Overskrive lokale variabler — flagg, autentiseringsstatus, peker. Forandrer programmets logikk uten å bytte hvilken kode som kjøres.
  2. Overskrive exception handler — peker til kode som kjører ved unntak. Angriper kan velge hva som kjøres hvis exception trigges.
  3. Overskrive returadressen — når funksjonen returnerer, hopper CPU-en til adressen angriper valgte. Total kontrollflyt-hijacking.
05 · AuthMe-angrepet

Bypass autentisering via overflow

Et klassisk eksempel: en lokal flag-variabel som overskrives av en buffer rett under.

Tenk en funksjon med en 4-byte buffer og en lokal int authenticated = 0. Etter passordsjekk settes flagget til 1 hvis riktig passord, ellers 0. Men funksjonen kopierer brukerinput til bufferen med strcpy. Angriperen sender inn en streng som er lengre enn 4 byte — overflow-en overskriver authenticated-flagget med vilkårlige bytes som ikke er 0. Senere sjekker koden if(authenticated) — som gir true, og angriperen får tilgang uten gyldig passord.

06 · Kjør angriperens kode

Returadresse-hijacking

Total kontrollflyt-overtakelse: overskriv returadressen, hopp dit angriperen vil.

Den klassiske «smashing the stack»-teknikken: angriperen plasserer shellcode (rå maskininstruksjoner) i selve bufferen, og overflødes returadressen så den peker tilbake inn i bufferen. Når funksjonen returnerer, hopper CPU-en inn i shellcoden og kjører den. Resultat: vilkårlig kjøring av angriperens kode med programmets rettigheter — typisk en remote shell.

Moderne varianter omgår grunnleggende forsvar:

  • Return-to-libc: i stedet for shellcode i bufferen, peker returadressen til eksisterende kode i libc, f.eks. system("/bin/sh"). Omgår «non-executable stack» (NX-bit).
  • ROP (Return-Oriented Programming): kjeder sammen små kodebiter («gadgets») som ender med ret. Bygger arbitrary computation av eksisterende kodebiter. Omgår både NX og signaturbasert deteksjon.
  • JOP/COP: varianter som bruker indirect jumps eller calls i stedet for ret.
07 · Heartbleed

Heartbleed: read overflow

Ikke alle overflow er skrive-overflow. Heartbleed lekket data ved å lese forbi bufferens slutt.

Heartbleed (CVE-2014-0160, OpenSSL) var en read overflow. SSL-protokollen har en «heartbeat»-melding der klient og server bekrefter at forbindelsen er aktiv. Klienten sender en payload og en lengde, og serveren skal ekko payloaden tilbake. Bug: serveren stolte på lengdefeltet uten å sjekke at det stemte med faktisk payload-størrelse.

// Forenklet — SSL-server
char buf[128];
// motta: client_payload + length-felt
for (int i = 0; i < length; i++) {
    putchar(buf[i]);   // echo tilbake
}

Angriperen sender en kort melding (1 byte) men hevder lengden er 65 535. Serveren leser 65 535 bytes fra minnet etter buf og sender dem tilbake. I praksis ble private SSL-nøkler, sesjonskoder og passord lekket fra millioner av webservere før patchen kom ut.

Hvorfor det er ille

Server-prosessen hadde tilgang til de private nøklene den brukte til TLS. Heartbleed lot enhver klient på internett lekke tilfeldig minne fra prosessen — og kunne gjentas til nøklene falt ut. Patch krevde at alle berørte servere både oppdaterte OpenSSL og roterte sertifikater og nøkler, fordi man ikke kunne vite hva som hadde blitt eksponert.

08 · Forsvar

Hvordan stoppe overflow

Forsvaret er flerlags: trygge funksjoner, kompilatorforsvar, statisk analyse, og helst typesikre språk.

Bytt ut usikre funksjoner:

UsikkerSikker erstatningMerk
strcpy(d, s)strcpy_s(d, n, s)Sjekker n, terminerer NULL
strcat(d, s)strcat_s(d, n, s)Samme
gets(buf)fgets(buf, n, stdin)Aldri bruk gets — fjernet i C11
sprintf(b, fmt, ...)snprintf(b, n, fmt, ...)Maks n bytes
strncpy(d, s, n)Med forsiktighetTerminerer ikke alltid med NULL — kan gi read overflow på strlen

strncpy-fellen: strncpy(str, "bye", 3) kopierer 3 bytes uten null. Etterfølgende strlen(str) leser videre i minnet til den finner en null — read overflow som lekker data.

Kompilatorforsvar:

  • GCC -fstack-protector: setter en «canary»-verdi mellom lokale variabler og returadressen. Sjekker at den er uendret før retur — ellers crasher prosessen.
  • MSVC /GS: tilsvarende stack canary i Visual Studio.
  • MSVC /SAFESEH, /SEHOP: beskytter exception handler-pekere mot overskrivelse.
  • ASLR (Address Space Layout Randomization): randomiserer stack/heap/library-adresser. Gjør det vanskelig å vite hvor man skal hoppe.
  • NX-bit / DEP: markerer stack/heap som ikke-kjørbar. Stopper klassisk shellcode-injeksjon.
  • Control Flow Integrity (CFI): verifiserer ved kjøretid at indirect calls/returns peker på lovlige mål.

Statisk analyse + fuzzing: verktøy som AFL, libFuzzer og clang static analyzer fanger overflow-mønstre i kildekoden eller via mutert input.

Typesikre språk: Python, Java, Go, Rust, C# — runtime sjekker grenser eller eier-modellen forhindrer ugyldig minneaksess. Ingen klassisk buffer overflow er mulig i sikker Rust eller Go uten unsafe-blokker. Microsoft, Google og Linux-kjernen migrerer i økende grad sikkerhetskritisk kode til Rust.

Lagdelt forsvar

Ingen enkelt forsvar er nok. -fstack-protector stoppes av kompilatorflagg-glipp; ASLR omgås ved info leak; NX omgås av ROP. Forsvar dybde: trygge funksjoner + kompilatorforsvar + statisk analyse + fuzzing + minneovervåkning + helst migrering til typesikkert språk for nye komponenter.
