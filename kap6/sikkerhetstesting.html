Kapittel 6 · Sikkerhetstesting

Sikkerhetstesting

Automatiserte verktøy og menneskelig analyse er komplementære — ingen av dem er tilstrekkelig alene.

01SAST 02DAST 03IAST 04Fuzz testing
01 · SAST

SAST

Static Application Security Testing analyserer kildekoden uten å kjøre applikasjonen — rask, integrert i IDE og CI.

SAST-verktøy bygger et Abstract Syntax Tree (AST) av koden og kjører analyser over det. Taint tracking følger data fra brukerinput (source) gjennom koden til potensielt farlige operasjoner (sink) — f.eks. fra request.GET['q'] til en SQL-spørring uten parameterisering. Hvis det finnes en flyt fra source til sink uten sanitering, rapporteres det som en sårbarhet.

VerktøyStyrkeSvakhet
SonarQubeBredt språkstøtte, integrering i IDE og CIHøy false positive rate uten kalibrering

Integrering i praksis:

  • Kjør SAST på hvert commit og PR — gi resultater som PR-kommentarer.
  • Blokker merge på kritiske og høy-alvorlighets funn med lav false positive rate.
  • Supprimer kjente falske positiver med annotasjoner i koden (// nosec, @SuppressWarnings).
  • Mål og forbedre false positive rate over tid.
02 · DAST

DAST

Dynamic Application Security Testing tester applikasjonen utenfra mens den kjører — simulerer en angriper uten kildekodekjennskap.

DAST-verktøy sender HTTP-forespørsler til applikasjonen, analyserer svarene og leter etter indikatorene på kjente sårbarhetsmønstre: XSS-payload i reflekterte svar, SQL-feilmeldinger ved injection, SSRF-forespørsler til interne adresser. DAST krever en kjørende staging-applikasjon og er tregere enn SAST.

Hva DAST finner som SAST ikke ser:

  • Sesjonshåndteringsfeil (manglende cookie-flagg, forutsigbare session-tokens).
  • Eksponerte endepunkter som ikke er dokumentert i koden (admin-ruter, debug-endepunkter).
  • Transport-sikkerhet (TLS-konfigurasjoner, HTTP-redirect-kjedene).
  • Kjøretids-XSS og CSRF som avhenger av applikasjonstilstand.

OWASP ZAP er det mest brukte open-source DAST-verktøyet. Det har en automated scan-modus for CI/CD og en interaktiv modus for manuell testing. Burp Suite er standarden for manuelle penetrasjonstester.

Merk

DAST kan ikke finne logiske tilgangskontrollfeil — det vet ikke at bruker A ikke skal se bruker Bs data. Det krever manuell testing eller business-logic-bevisst testing. Se Kapittel 2 (IDOR).

03 · IAST

IAST

Interactive Application Security Testing kombinerer statisk og dynamisk analyse ved å instrumentere applikasjonen under kjøring.

IAST installerer en agent i applikasjonens runtime som overvåker kodekjøring mens funksjonelle tester kjøres. Agenten ser både input (fra tester) og kodeflyt (intern) — og kan identifisere sårbarheter med langt lavere false positive rate enn SAST, fordi den vet nøyaktig hvilken kode som faktisk kjøres under hvilke forutsetninger.

Fordelen: høy nøyaktighet og lave false positiver. Ulempen: krever en agent i applikasjonsruntimen, som ikke alltid er mulig (microservices med mange språk, produksjonsmiljøer). Passer best i test-/staging-miljøer der funksjonelle tester kjøres.

04 · Fuzz testing

Fuzz testing

Fuzzing genererer inndata automatisk — og finner feil ingen tenkte på å teste for manuelt.

En fuzzer sender enorme mengder tilfeldig eller semi-tilfeldig inndata til et system og observerer om det krasjer, henger, lekker minne eller oppfører seg uventet. Coverage-guided fuzzing (AFL++, libFuzzer) instrumenterer koden for å måle hvilke kodestier inputen tar — og genererer nye inputs designet for å treffe ubesøkte stier. Google har brukt dette til å finne tusenvis av sårbarheter i open-source kode via OSS-Fuzz.

# Strukturert fuzzing med Hypothesis (Python)
from hypothesis import given, strategies as st

@given(st.text())
def test_parser_not_crash(input_str):
    # Parseren skal ikke krasje uavhengig av input
    try:
        parse_config(input_str)
    except ValueError:
        pass  # Forventet feil er ok
    # Uventet exception her = sårbarhet

Fuzzing er spesielt effektivt for:

  • Parsere (JSON, XML, binærformater, protokoller).
  • Kompressorer og kryptobiblioteker.
  • Input-behandling i nettapplikasjoner (filoppasting, API-endepunkter).

Oppsummering

Ingen enkelt testmetode finner alt. SAST finner implementeringsfeil tidlig. DAST finner kjøretidsfeil utenfra. IAST kombinerer begge med høy nøyaktighet. Fuzzing finner det ingen tenkte på. Manuell penetrasjonstest finner logiske feil. Bruk dem lagvis.
