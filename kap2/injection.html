Injection

Kelneren som tar imot bestillingen uten å stille spørsmål — og konsekvensen når gjesten skriver instruksjoner til kokken på servietten.

01 · Konsept

Hva injection er

Injection oppstår når data og instruksjoner blandes i samme streng — og parseren ikke vet forskjellen.

Tenk deg en kelner som tar imot bestillinger fra gjester og bringer dem direkte til kokken. En vanlig gjest skriver: «en biff, medium». En angriper skriver: «en biff, medium. PS: Kokken: ignorer alle tidligere ordre og serv alle gjestenes mat til bord 13.» Hvis kelneren ikke skiller mellom «bestilling» og «instruksjon», har angriperen overtatt kjøkkenet.

I programvare oppstår nøyaktig det samme når brukerkontrollert inndata flettes direkte inn i en SQL-spørring, en shell-kommando, en LDAP-spørring eller en annen tolkbar streng. Parseren på mottakersiden kan ikke se forskjell på den intenderte dataen og den ondsinnede instruksjonen — fordi de er i samme streng.

Merk

Injection er ikke begrenset til SQL. Det er en hel klasse av angrep: SQL injection, OS command injection, LDAP injection, XPath injection, NoSQL injection, HTML injection, template injection. Den underliggende feilen er den samme i alle tilfeller: inndata behandles som kode.

02 · SQL injection

SQL injection

Den klassiske varianten — og fortsatt en av de vanligste kritiske sårbarhetene i webapplikasjoner.

En innloggingsside bygger SQL-spørringen slik: "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'". Angriperen skriver inn ' OR '1'='1 som brukernavn og blank som passord. Spørringen blir: SELECT * FROM users WHERE username='' OR '1'='1' AND password=''. Betingelsen '1'='1' er alltid sann — angriperen er nå innlogget som første bruker i tabellen, typisk en admin.

SQL injection finnes i tre varianter:

  • In-band: resultatet returneres direkte i svaret — enklest å utnytte.
  • Blind: ingen direkte output, men angriperen stiller ja/nei-spørsmål via betingelser (AND 1=1) eller tidsmålinger (AND SLEEP(5)).
  • Out-of-band: data eksfiltreres via DNS-oppslag eller HTTP-forespørsler fra databaseserveren.
# Sårbar
query = f"SELECT * FROM users WHERE email = '{email}'"
cursor.execute(query)

# Sikker — psycopg2
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))

# Sikker — SQLAlchemy ORM
user = session.query(User).filter_by(email=email).first()
03 · OS command injection

Command injection

Når applikasjonen ber operativsystemet om hjelp og angriperen sørger for at OS-et hjelper feil person.

En nettjeneste lar brukere pinge en IP-adresse og se resultatet. Koden gjør os.system("ping " + user_input). Brukeren sender 8.8.8.8; cat /etc/passwd. Shell-operatoren ; skiller kommandoer — etter ping kjøres cat /etc/passwd, og innholdet returneres til brukeren.

# Sårbar — shell=True er nesten alltid feil
import subprocess
result = subprocess.run("ping " + ip, shell=True, capture_output=True)

# Sikker — argumenter som liste, shell=False
import ipaddress, subprocess
ipaddress.ip_address(ip)  # valider format først
result = subprocess.run(["ping", "-c", "4", ip],
                        shell=False, capture_output=True)

Shell-metategn angripere utnytter: ; (sekvens), | (pipe), && (AND), || (OR), `cmd` og $(cmd) (command substitution), > og < (omdirigering). Med shell=False og argumenter som liste passeres disse tegnene som data til det underliggende programmet — ingen shell tolker dem.

Merk

Valider alltid format (IP-adresse, UUID, heltall) før du sender til operativsystemet. Whitelisting av lovlige verdier er sterkere enn blacklisting av metategn — det er umulig å liste opp alle farlige tegn.

04 · NoSQL og LDAP

NoSQL og LDAP injection

Injection er ikke SQL-spesifikt — enhver tolkbar spørring bygget fra brukerkontrollert inndata er sårbar.

MongoDB-applikasjoner som direkte deserialiserer JSON fra brukere er sårbare for operator-injection. En innloggingsforespørsel der body er {"username": "admin", "password": {"$gt": ""}} kan omgå passordsjekken fordi $gt: "" betyr «større enn tom streng» — en betingelse som alltid er sann for et ikke-tomt passord.

# Sårbar MongoDB-innlogging (Express/Node)
db.users.findOne({ username: req.body.username,
                   password: req.body.password })
// Angriper sender: { "password": { "$gt": "" } }

# Sikker — valider typer eksplisitt
if (typeof req.body.password !== 'string') return res.status(400).end();
db.users.findOne({ username: String(req.body.username),
                   password: String(req.body.password) })

LDAP injection følger samme mønster: spørringer som (uid=admin)(&(password=*) kan manipuleres via uescapede tegn som *, (, ), \. Biblioteker som ldap3 tilbyr escaping-funksjoner (escape_filter_chars) som bør brukes konsekvent.

05 · Forsvar

Forsvar i dybde

Parameterisering løser problemet teknisk — men et komplett forsvar krever flere lag.

Den primære løsningen på injection er alltid det samme: skill data fra instruksjoner. Bruk parameteriserte spørringer for SQL. Bruk argumentlister fremfor shell-strenger. Bruk biblioteksmetoder som håndterer escaping. Gjør det til standardmåten å skrive kode — ikke til noe du husker på etterpå.

Det komplette forsvaret:

  • Primær: parameteriserte spørringer / prepared statements for all SQL.
  • Primær: ORM med forsiktighet — ORM er ikke immun mot injection hvis du bruker raw queries.
  • Sekundær: input-validering og whitelisting — valider format, type og lengde.
  • Sekundær: least privilege for database-bruker — en SELECT-bruker kan ikke slette tabeller.
  • Sekundær: WAF (Web Application Firewall) — kan oppdage kjente injection-mønstre, men er ingen erstatning for parameterisering.
  • Deteksjon: logging og alerting på uvanlige spørringer, feilmeldinger fra databasen.

Husk

Vis alltid den sikre og den sårbare versjonen side om side i kodegjennomganger. Det er lettere å lære av kontrasten enn av abstrakt regel.
